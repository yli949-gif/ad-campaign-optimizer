Safety guardrails in force

  • Pausing always needs human approval — even in fully autonomous mode, the agent never pauses a campaign on its own.
  • Single budget moves capped at ±25% — no campaign's budget can swing more than a quarter in one run.
  • Account budget ceiling — total automatic budget increases are capped (≈+20%) to prevent runaway spend; anything beyond is held for approval.
  • New campaigns launch paused — drafted creative and settings wait for a person to enable them.
  • Low-signal campaigns left alone — the agent won't act on too few conversions to read reliably.

Designed for future MCP integration — how this connects to live ad platforms later

Today
Runs fully offline on simulated data — no API keys, no installs.
The seam
Every platform call goes through one interface (platforms.py). Swapping mock for live is a single, contained change.
Read tools
get_campaigns / get_performance map onto Synter Media, the official Google Ads MCP, or Meta/Pipeboard.
Write tools
update_budget / update_bid / create_campaign map onto the same MCP servers' write actions.
Creative
The image prompts route to Imagen 4, Flux, or Runway via a text-to-image tool.
Recommended rollout
Keep dry_run as default for the first weeks, review proposals, then grant assist and finally auto.

Limitations

  • Offline mock data. Every number here is simulated for demonstration. It is not connected to any live ad account and reflects no real spend or performance.
  • Not production-ready. This is an MVP that shows the agentic workflow and safety model — it is designed for, but not yet wired to, live MCP integrations.
  • No guaranteed outcomes. "Estimated wasted spend" and "potential reallocation" are illustrative indications, not promised savings.
  • Transparent rules, not a learned model. Decisions come from simple, explainable rules; a real deployment would validate them against live results before granting autonomy.
